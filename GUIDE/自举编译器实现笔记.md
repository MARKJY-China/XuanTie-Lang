# 自举编译器实现笔记

> 本文档是玄铁自举编译器（`xuantie_compiler/` 目录下由玄铁自身编写的编译器）的实现笔记，面向未来的维护者和协作者。
> 自举编译器由 Gemini 辅助开发完成主体架构，Claude 辅助种子编译器优化。本笔记基于 2026-05-03 实际代码审计编写。

---

## 1. 编译器总览与架构

### 1.1 编译流程

```
玄铁源码 (.xt) → 词法.xt → 语法分析.xt → 抽象语法树.xt → 编译.xt（四趟）→ LLVM IR → clang/gcc → 原生二进制
```

自举编译器是一个用**玄铁自身编写**的四趟编译器，核心文件：

| 文件 | 行数 | 职责 |
|------|------|------|
| `词法.xt` | 462 | 词法分析，将 UTF-8 字符转为 Token 流 |
| `抽象语法树.xt` | 478 | 所有 AST 节点类型定义 |
| `语法分析.xt` | 1107 | 递归下降 + Pratt Parsing 语法分析 |
| `编译.xt` | 3187 | **核心**：四趟编译——预扫描、语义分析、类型推导、IR 生成 |
| `玄铁.xt` / `xt.xt` / `入口.xt` | 110-222 | 驱动入口，CLI 参数解析、调用编译器、执行 clang/gcc |

### 1.2 与种子编译器的关键差异

| 维度 | 种子编译器 (Go) | 自举编译器 (玄铁) |
|------|----------------|-------------------|
| 编译趟数 | 单趟（代码生成即类型追踪） | 四趟（预扫描→语义→类型推导→IR） |
| 类型追踪 | 内联在 `compileExpression` 返回值 | 独立的 `推导程序类型` pass + `变量类型表` |
| 前向引用 | AST 节点天然支持 | 通过全局预扫描收集符号解决 |
| 模块系统 | `ImportExpression` 内联处理 | 全局预扫描递归 + `程序缓存` + `语义分析` 拓扑排序 |
| 运算符重载 | `InfixExpression` + `magicMethod` fallback | `编译中缀` 先试 `xt_dict_get` 动态分发，再 fallback `编译内建中缀` |
| 内存管理 | Go GC | Arena 内存池（`xt_arena_new`/`xt_arena_use`） |

### 1.3 编译器结构体关键字段

出自 [编译.xt L1-L40](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt)：

```xuantie
型 编译器 {
    设 结果输出列表   // 当前函数/主域的 IR 行列表
    设 结果输出       // 最终拼接的完整 IR 字符串
    设 寄存器数       // 递增计数器，生成 %t1, %t2...
    设 标签数         // 递增计数器，生成 if.then.1, loop.cond.2...
    设 字符串映射     // 字符串常量去重 → 全局别名映射
    设 作用域栈       // [[addrReg...]]，每层作用域需要 ARC 释放的寄存器列表
    设 符号表         // 变量名 → alloca 寄存器地址或全局符号名
    设 变量类型表     // 变量名 → 类型（"raw_i64" / "i64" / "i1" / "型:类名"）
    设 函数表         // 函数名 → 真，标记已声明的函数
    设 类型表         // 类型名 → AST 节点，存储型定义
    设 函数输出       // 所有函数定义的 IR 文本
    设 外部函数输出列表 // 外部函数声明的 IR 行列表
    设 当前型名称     // 编译型定义方法时的上下文
    设 是全局         // 真 = 在主域，假 = 在函数/方法内
    设 已访引入       // 防止重复引入
    设 程序缓存       // 文件路径 → AST 程序节点，避免重复解析
    设 已访语义       // 防止语义分析死循环
    设 已访别名       // 追踪模块别名，防止被误当作函数
    设 主文件路径     // 用于相对路径解析
    设 断标签栈       // break 目标标签栈
    设 续标签栈       // continue 目标标签栈
}
```

## 2. 词法分析器（词法.xt）

位于 [词法.xt](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/词法.xt)，462 行。

### 2.1 Token 结构

```xuantie
型 标记 {
    设 标记类型 = ""   // Token 类型关键字，如 "标识符"、"数字"、"函数定义"
    设 标记字面量 = "" // 原始文本
    设 标记行 = 0      // 行号（1-based）
    设 标记列 = 0      // 列号（1-based）
}
```

### 2.2 词法分析器工作机制

**字符流模型**：将源码字符串转为字符列表 `字符列表`，通过 `读字符()` 单字符前进，`窥视字符()` 预读下一字符。

**关键字识别**（[词法.xt L409-L464](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/词法.xt#L409)）：所有中文关键字通过硬编码匹配表识别。例如：
```
"设" → "变量声明"、"函" → "函数定义"、"型" → "型定义"
"若" → "如果"、"遍历" → "遍历循环"、"匹配" → "匹配"
```

**字符串处理**（[词法.xt L230-L296](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/词法.xt#L230)）：
- 支持 `"..."` 和 `'''...'''` 两种引号风格
- 支持插值标记 `#{}`——词法器**不解析插值**，将整段字符串（含 `#{表达式}` 原文）作为一个 Token 返回，插值解析在语法分析阶段处理

**标识符规则**（[词法.xt L88-L114](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/词法.xt#L88)）：
- 排除所有 ASCII 符号和数字后，剩余字符（含所有多字节 UTF-8 字节）视为"字母"
- 支持以 `?` 结尾（如 `存在?`、`包含?`）

**注释**：`//` 到行尾，在 `读标记()` 中处理。

## 3. 抽象语法树（抽象语法树.xt）

位于 [抽象语法树.xt](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/抽象语法树.xt)，478 行。

### 3.1 基础节点

```xuantie
型 基础节点 {
    设 节点行       // 源码行号
    设 节点列       // 源码列号
    设 节点类别     // "函数定义"、"中缀"、"标识符" 等——编译器的核心分发依据
    设 推导类型     // 第三趟类型推导的结果（"整"、"字"、"逻"、"未知"）
}
```

**关键设计**：所有节点继承 `基础节点`，`节点类别` 是编译器 `匹配` 语句的**唯一分发键**。这是理解整个编译器控制流的核心——不是靠 Go 的类型 switch，而是靠字符串匹配。

### 3.2 关键节点类型速查

| 节点类型 | 节点类别 | 关键字段 |
|---------|---------|---------|
| `整数节点` | `"整数"` | `整数值` |
| `标识符节点` | `"标识符"` | `标识符名称` |
| `中缀表达式节点` | `"中缀"` | `中缀左`、`中缀运算符`、`中缀右` |
| `前缀表达式节点` | `"前缀"` | `操作符`、`右节点` |
| `调用表达式节点` | `"调用表达式"` | `函数表达式`、`调用参数` |
| `成员访问节点` | `"成员访问"` | `左表达式`、`成员名称` |
| `函数定义节点` | `"函数定义"` | `函数名`、`参数列表`、`返回类型`、`函数体块` |
| `变量声明语句节点` | `"变量声明"` | `声明名称`、`声明值`、`注解类型` |
| `块语句节点` | `"块"` | `块语句列表` |
| `若语句节点` | `"若语句"` | `若条件`、`若结果块`、`抑分支列表`、`若否则块` |
| `遍历循环节点` | `"遍历循环"` | `迭代变量名`、`迭代目标`、`循环块` |
| `引入表达式节点` | `"引入"` | `路径`、`别名` |

### 3.3 函数定义节点的参数表示

**重要**：在自举编译器的 AST 中，函数参数用 `标识符节点` 的列表表示，类型注解存在 `标识符节点.推导类型` 字段中（[语法分析.xt L502](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/语法分析.xt#L502)）：

```xuantie
p节点.推导类型 = 此.解析类型()  // 存的是原始类型名，如 "整"、"字"
```

这与种子编译器（Go）的 `Parameter.DataType` 不同——自举编译器**复用了 AST 节点上已有的 `推导类型` 字段**来存储参数类型注解，而不是新增一个专门的字段。

流程：
1. 解析函数名（允许 `造` 作为构造函数名）
2. 解析泛型参数 `<T>`
3. 解析参数列表 `(name: Type, ...)`
4. 解析返回类型注解 `: Type`
5. 解析函数体 `{ ... }`

**类型注解处理**：每个参数的 `推导类型` 字段被重用存储解析到的类型名（如 `"整"`），这是后续类型推导和代码生成的关键数据。

## 5. 编译器（编译.xt）—— 四趟编译详解

位于 [编译.xt](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt)，3187 行，是自举编译器最大的文件。

### 5.1 编译入口：编译程序

```xuantie
函 编译程序(程序节点, 源文件路径) {
    // 第一趟：全局预扫描（收集符号）
    此.全局预扫描(程序节点, "", 源文件路径)
    // 第二趟：语义分析（类型成员解析 + 拓扑排序）
    此.语义分析(程序节点)
    // 第三趟：类型推导（Static Type Inference）
    此.推导程序类型(程序节点)
    // 第四趟：代码生成（IR Emit）
    // ...遍历程序语句列表，调用 编译语句()...
    // 最后拼接 IR 头部（类型声明、外部函数声明、字符串常量）+ 函数体 + main
}
```

### 5.2 第一趟：全局预扫描

函数位置：[编译.xt L239](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L239)。

**职责**：在代码生成前收集所有顶层符号，解决前向引用问题。

**匹配的节点类别**：
- `"函数定义"` → 注册到 `函数表` 和 `符号表`
- `"型定义"` → 注册到 `类型表`，预扫描所有方法签名
- `"接口定义"` → 注册到 `类型表`
- `"外部函数"` → 调用 `编译外部函数定义` 生成声明
- `"表达式语句"` 且是 `"引入"` → 递归预扫描引入的模块（`预扫描引入`）
- `"变量声明"` → 全局变量注册到 `符号表`

**引入机制**：[预扫描引入](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L300)：
1. 按主文件路径解析相对路径
2. 检查 `程序缓存`——若已解析过则复用 AST
3. 否则 `文件.读` + `造 词法分析器` + `造 语法分析器` + `解析程序()`
4. 将解析结果缓存到 `程序缓存`
5. 递归调用 `全局预扫描`，带上命名空间前缀

### 5.3 第二趟：语义分析

函数位置：[编译.xt L319](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L319)。

**职责**：解析类型成员、处理继承链、插入 `此` 参数。

流程：
1. 递归处理引入模块的语义分析
2. 对型定义做拓扑排序（先解析父类，再解析子类）
3. `解析型成员` 中：
   - 继承父类的属性和方法列表（扁平化到 `最终属性列表` / `最终方法列表`）
   - 为每个方法的参数列表**插入 `此` 参数**（如果没有的话）
   - 处理方法重写：用 `原始函数名` 匹配，覆盖同名的父类方法

**关键细节**：语义分析修改了 AST 节点——给方法加了 `此` 参数后，函数定义的 `参数列表` 变化了。这影响后续类型推导和代码生成的参数遍历。

### 5.4 第三趟：类型推导

函数位置：[编译.xt L3405](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L3405)。

**职责**：遍历 AST 树，为每个表达式节点打上 `推导类型` 标签。

**推导规则**：
- 字面量：`"整数"` → `"整"`、`"小数"` → `"小数"`、`"字符串"` → `"字"`
- 标识符：查 `变量类型表`
- 中缀表达式：加减乘除 → `"整"` 或 `"小数"`；比较/逻辑 → `"逻"`；连接 → `"字"`
- 调用表达式：查函数返回值类型（`变量类型表`）
- 成员调用：`.字节`、`.长度`、`.字符数` → `"整"`；`.包含?`、`.成功` → `"逻"`

**用途**：代码生成阶段的 `编译内建中缀` 会读取 `节点.中缀左.推导类型 == "整"` 来决定是否走原生指令路径。

### 5.6 代码生成 —— compileExpression

位于 [编译.xt L1471](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L1471)。返回 `[regIR, type]` 二元组。

**关键路径**：

**标识符** → 查 `符号表` 找到 alloca 地址 → `load i64` → 查 `变量类型表` 获取类型 → 标量类型跳过 retain：

```xuantie
若 非 此.是标量类型(类型) {
    此.写出("  call void @xt_retain(i64 " & 寄存器 & ")")
}
返 [寄存器, 类型]
```

**中缀表达式** → 先试运算符重载（`编译中缀` L2974），无重载则走 `编译内建中缀`（L3045）：

`编译内建中缀` 有**双重路径**：
1. **静态优化路径**：`节点.中缀左.推导类型 == "整"` 且 `节点.中缀右.推导类型 == "整"` → 原生 LLVM 指令（`add`/`sub`/`mul`/`icmp`...）
2. **通用路径**：`xt_eq`、`xt_compare` 运行时函数调用

**调用表达式** → `编译调用表达式`（L2895+）：识别 `printf` 的特殊处理（`xt_ffi_printf`），其余走 `执行动态调用`（函数指针 `inttoptr` + `call`）。

### 5.7 关键辅助函数

| 函数 | 作用 |
|------|------|
| `符(值)` | 将任意值安全转为 IR 字符串，遇到 `空` 返回 `"%NULL_SYMBOL_ERR"` 防止污染 IR |
| `标(值)` | 同 `符`，用于标签 |
| `确保I64` | 将任意类型值转为标记 i64（raw_i64→装箱、i1→select、函数名→ptrtoint） |
| `确保RawI64` | 将标记 i64 脱壳为原始 i64（`ashr i64 %x, 1`） |
| `是标量类型` | 判断 `raw_i64`/`i1`/`double` |
| `写_释放_xt` | 按类型释放：标量→无操作，i64→`xt_release` |
| `写_存储_xt` | 按类型存储到 alloca 地址：标量存原值，i64 存标记值 |
| `生成布尔条件` | 将任意类型转为 i1 条件值 |
| `最后是返回` | 检查输出列表最后一行是否是 `ret`/`br`/`unreachable` |

### 5.8 IR 输出与主函数组装

[编译程序](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L416) 的末尾按顺序输出：
1. Target triple 和 data layout
2. 类型声明（`%XTObject`、`%XTString`、`%XTArray`……）
3. 外部运行时函数声明（`declare ... @xt_...`）
4. 外部 FFI 函数声明
5. 字符串常量（`@str.N = ... constant ...`）
6. 函数定义（`此.函数输出`）
7. `define i32 @main` 入口

**注意**：自举编译器的类型声明与种子编译器略有不同——自举的 `%XTObject` 缺少 `magic` 字段（只有 `{ i32, i32 }` 而非 `{ i32, i32, i32 }`）。这是 INVESTIGATION_xt.md 中提到的 `编译.xt` 类型定义与运行时不一致的问题之一，但只有用自举编译器编译出的**产物**运行时才会受影响，自举编译器本身由种子编译器编译不受影响。

## 6. 驱动入口（玄铁.xt / xt.xt / 入口.xt）

自举编译器的入口有两个版本：
- **玄铁.xt**（222 行）：含 Arena 内存池 + 完整的 clang/gcc 调用流程，是正式版本
- **xt.xt**（110 行）：使用 `编译.生成原生程序()` 封装的简化版本

### 6.1 命令行流程（玄铁.xt）

```
解析参数 → 创建 Arena(10MB) → 文件.读 → 造词法器 → 造语法分析器 → 解析程序
  → 造编译器 → 编译.编译程序 → 文件.写(IR) → clang -c → gcc 链接 → 清理
```

**关键：** 玄铁.xt 默认使用 `-Og` 优化等级（[L34](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/玄铁.xt)），而非 `-O2`。且优化等级变量名是 `优化等级` 而非 `优化等级_str`，存储如 `"-Og"` 这样的字符串。

## 7. 已知陷阱与隐形契约

### 7.1 【严重】compileFunctionDefinition 未读取参数类型注解

位置：[编译.xt L1197](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L1197)

```xuantie
若 arg节点2.标识符名称 == "此" 且 此.当前型名称 != "" {
    此.变量类型表[arg节点2.标识符名称] = "型:" & 此.当前型名称
} 否 {
    此.变量类型表[arg节点2.标识符名称] = "i64"    // ← 永远是 "i64"
}
```

**影响**：即使源码写了 `函 xxx(n: 整): 整`，所有参数仍被标记为 `"i64"`（标记值），不会走 raw_i64 优化。这解释了为什么当前自举编译器编译 `斐波那契(n: 整): 整` 和 `斐波那契(n)` 耗时几乎相同（约 365ms vs 400ms）。

**修复方向**：应类似种子编译器的 `mapTypeAnnotation` 逻辑，读取 `arg节点2.推导类型`（即解析器存入的 `"整"`/`"小数"`/`"逻"`），映射为 `"raw_i64"`/`"double"`/`"i1"`。

### 7.2 【严重】编译赋值未处理标量变量存储

位置：[编译.xt L3181](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L3181)

```xuantie
此.写出("  store i64 " & xt_val & ", i64* " & 存址)
```

`xt_val` 是 `确保I64` 后的标记值。如果变量类型是 `raw_i64`，应该存储 `valReg`（原始值）。这个 bug 与种子编译器之前修复的 AssignStatement 标量存储 bug 是同源的——会导致 `k = k + 1` 后 `k` 变成标记值，后续 `k < N` 判断失效。

### 7.3 推导程序类型的结果未被代码生成充分利用

第三趟 `推导程序类型` 给每个 AST 节点打上了 `推导类型`（`"整"`/`"字"`/`"逻"`），但在代码生成阶段只有 `编译内建中缀`（L3050）读取 `节点.中缀左.推导类型 == "整"` 来做静态优化。`编译函数定义` 完全无视了类型推导的结果——参数类型从 `变量类型表` 读取，而 `变量类型表` 在函数入口处**不读** `arg节点.推导类型`，全部硬编码 `"i64"`。

### 7.4 形参类型注解存储在标识符节点的推导类型字段

语法分析器 [L502](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/语法分析.xt#L502) 中：
```xuantie
p节点.推导类型 = 此.解析类型()
```

这与种子编译器 `Parameter.DataType` 专用字段不同。自举编译器**复用**了 `推导类型` 字段来存原始类型名。这意味着第三趟 `推导程序类型` pass **可能会覆盖**参数上已经存好的类型注解——如果该标识符节点同时被类型推导遍历到的话。但目前没有观察到覆盖，因为推导 pass 不会遍历函数参数节点（只遍历语句和表达式）。

### 7.5 变量类型表的生命周期

`变量类型表` 是全局共享的，但在 `编译函数定义` 中会被**临时清空**并只恢复全局变量：

```xuantie
此.变量类型表 = {}
遍历 键 于 之前的符号表 {
    若 存址.截取(0, 1) == "@" { // 全局符号
        此.变量类型表[键] = 之前的类型表[键]
    }
}
```

这意味着**局部变量类型不会跨函数传播**——这是正确的行为。但如果将来需要函数级别的类型签名信息（如跨函数优化），需要额外的元数据记录。

### 7.6 作用域栈的两阶段退出

[出作用域](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L603) 中，变量按 `追踪对象()` 记录的顺序释放（FIFO），没有 Arena 池变量的优先/延后处理（和种子编译器的 `arenaPoolRegs` 不同）。

### 7.7 符() 和 标() 的空值保护

这两个包装函数（L198-L212）用于防止编译器内部状态异常（如 `空` 对象被误序列化为字符串）导致的 IR 污染。如果变量名是 `空`，会返回 `"%NULL_SYMBOL_ERR"` 这种显眼的错误标记，而不是静静地生成 `i64 空` 这样的非法 IR。

### 7.8 编译中缀的静态优化路径依赖推导类型

[编译内建中缀 L3050](file:///g:/FUsed_Folders/Project/XuanTie/xuantie_compiler/编译.xt#L3050)：

```xuantie
若 节点.中缀左.推导类型 == "整" 且 节点.中缀右.推导类型 == "整" {
```

这里比较的是第三趟推导 pass 打上的 `推导类型`（值为 `"整"`），而不是实际的 IR 类型（`"raw_i64"` vs `"i64"`）。如果某个整数变量因为类型注解缺失导致 `推导类型` 是 `"未知"`，即使运行时它确实是整数，这个静态优化路径也不会被触发。

### 7.9 UAF 崩溃（玄铁.xt 主函数）

来自 INVESTIGATION_xt.md：`玄铁.xt` 的 `主()` 函数在手动 `xt_arena_destroy(池)` 后，函数末尾编译器自动插入的 ARC 释放代码仍尝试 `xt_release(池)`，导致 Use-After-Free。

## 8. 代码生成 IR 示例

以 `函 斐波那契(n: 整): 整 { 若 n <= 1 { 返 n } 返 斐波那契(n-1) + 斐波那契(n-2) }` 为例，自举编译器**当前**（未修复 7.1）生成的 IR：

```llvm
define i64 @"斐波那契"(i64 %"this_arg", i64 %"n_arg") {
entry:
  %"n" = alloca i64
  store i64 %"n_arg", i64* %"n"
  call void @xt_retain(i64 %"n_arg")      ; ← 参数被当堆对象

  ; n <= 1 → xt_compare 运行时调用
  %t3 = call i32 @xt_compare(i8* %tObj1, i8* %tObj2)
  %t4 = icmp sle i32 %t3, 0

  ; n - 1 → xt_sub 运行时调用
  %t9 = call i8* @xt_sub(i8* %tObj3, i8* %tObj4)

  ; 递归调用 → 标记值传递
  %t11 = call i64 @"斐波那契"(i64 0, i64 %t10)
  call void @xt_release(i64 %t10)
```

**对比**：修复参数类型注解后应生成的 IR（参考种子编译器优化后）：

```llvm
define i64 @"斐波那契"(i64 %"n_arg") {
entry:
  %"n" = alloca i64
  %t1 = ashr i64 %"n_arg", 1          ; 参数脱壳
  store i64 %t1, i64* %"n"            ; 存 raw_i64
  ; 无 retain

  ; n <= 1 → icmp sle 原生指令
  %t2 = icmp sle i64 %t1, 1

  ; n - 1 → sub 原生指令
  %t8 = sub i64 %t1, 1

  ; 递归调用 + 脱壳
  %t10 = call i64 @"斐波那契"(i64 %t9)
  %t11 = ashr i64 %t10, 1             ; 返回脱壳
  ; ... add i64, shl+or 装箱，ret
```

## 9. 与种子编译器的功能差距总结

| 功能 | 种子编译器 | 自举编译器 | 影响 |
|------|----------|-----------|------|
| 参数类型注解读取 | ✅ 已修复（本轮对话） | ❌ 7.1：全部硬编码 "i64" | fib 无优化，~360ms |
| 返回值类型脱壳 | ✅ 已修复（本轮对话） | ❌ 未实现 | 调用方仍走运行时函数 |
| 标量赋值存储 | ✅ 已修复（本轮对话） | ❌ 7.2：存标记值 | 循环计数器 bug |
| 函数内 raw 运算 | ✅ 完整 | ✅ 有，但只对推导类型为 "整" 的生效 | 部分生效 |
| 跨函数 raw 通信 | ❌ ABI 限制 | ❌ ABI 限制 | 递归调用边界装箱不可避免 |
| IR 类型声明一致性 | ✅ 正确 | ❌ 7.8：%XTObject 缺 magic | 自举产物可能运行时崩溃 |
