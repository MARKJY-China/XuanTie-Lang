## 玄铁类型系统哲学设计

是由两套方案组成：

1. 多档开关——在执行编译指令时加入指令参数`--严格、--禁止、--默认`  即可切换编译器语法检查等级，具体分级制度如下
   - 错误等级：
     - 宽松级：未使用的变量　 | 　静默处理
     - 警告级：隐式类型、若抑否逻辑表达式出现多个 `否` 编译器会只采用第一个否、隐式动态、可推断未声明　 | 　编译期输出警告，强烈建议开发者修复这个问题
     - 错误级：完全错误的语法　 |　编译器完全退出并中断编译任务
   * 默认：按原分级处理（宽松→静默，警告→提示，错误→中断）
   * 严格：所有等级提升一档（宽松→警告，警告→错误，错误→错误）
   * 禁止：所有非错误提升为错误（宽松→错误，警告→错误，错误→错误）
2. 项目配置文件——在项目根目录放一个 `玄铁.配置.toml`
   ```
   [编译]
   严格度 = 禁止
   警告即错误 = ["隐式类型", "未使用变量"]
   忽略警告 = ["注释格式"]

   [版本]
   玄铁版本 = "0.18"
   ```
  当配置和命令行冲突时怎么办？请看配置优先级
   - 配置优先级（从高到低）：
     - 命令行参数 > 项目配置文件 > 编译器默认值
     - 当命令行参数显式填写了严格声明，那么就使用严格模式编译
  `"隐式类型"` 究竟指什么？变量没标注？参数没标注？返回值没标注？
   - **"警告码"机制**：
     ```
     警告 W001 - 隐式类型（变量未标注）            [宽松]
     警告 W002 - 隐式类型（参数未标注）            [警告]
     警告 W003 - 隐式类型（返回值未标注）          [警告]
     警告 W004 - 逻辑语法（多个"否"分支）          [警告]
     警告 W005 - 弱引用语法（缺少可空修饰 ?）       [警告]
     警告 W006 - 路径规范（斜杠混用）              [警告]
     警告 W101 - 未使用变量                        [宽松]
     警告 W102 - 未使用导入                        [宽松]
     警告 W201 - 注释格式不规范                    [宽松]
     警告 W401 - 循环引用（强引用环）              [警告]
     ```
   - 配置文件里可以用警告码或类别名：
      ```
      警告即错误 = ["W001", "W002"]    # 精确控制
      # 或
      警告即错误 = ["隐式类型"]         # 类别一键控制（包含所有 W00X）
      ```
  怎么在源码里局部抑制警告？
   - **注解式**：在变量前添加 `@允许(未使用变量)` 即可抑制警告

    // 注解式（跟 Rust/TS 一致）
    @允许(未使用变量)
    设 x = 5
    
3. 等级可见可对话 
警告输出里要带等级信息：
```
警告 [warning-implicit-type] (严格度 严格)：第 23 行
  参数 n 缺少显式类型注解
  推断结果：整
  修复建议：函 斐波那契(n: 整): 整 { ... }
  这个警告在严格度 [严格] 下会成为错误。
```
