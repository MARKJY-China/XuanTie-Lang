> - [X] **已实现**：语法已解析，IR 已生成，且在 `入口.xt` 的自举测试中稳定运行。

## 8. 编译器架构演进与未来规划 (Future Evolution)

- [X] 多趟编译与语义分析 (Multi-pass Compilation) —— *注：已引入独立的预扫描与语义分析阶段，完美支持前向引用与多模块符号隔离*

- **静态类型推导与去虚化 (Devirtualization)**：在编译期推断对象类型。如果确认为基础类型，则直接生成 LLVM 原生运算指令（如 `fadd`），绕过动态分发（魔术方法），大幅提升性能。
- **现代化错误报告 (Diagnostics & DX)**：实现带颜色高亮、具体错误位置（行列号）及修复建议的编译错误提示（类似 Rust）。循环引用警告格式示例：

  ```
  ⚠ 玄铁编译警告 [循环引用]
     → 第 12 行：节点.下一个
     → 第 18 行：节点.父节点
     已自动将 '父节点' 降级为弱引用
     如需手动控制，请使用 '弱 设' 显式声明
  ```
- **模块化构建系统 (Build System)**：完善 `引` 机制，支持多文件、循环依赖解析，最终演化出玄铁专属的包管理与构建工具 (TiePM)。
- **跨平台与 FFI 增强**：完善对 C 动态库的无缝调用（`外`），并利用 LLVM 完善对 Windows/Linux/macOS 及 WebAssembly 的交叉编译支持。
- **调试信息 (Debug Info)**：在生成的 LLVM IR 中嵌入 DWARF 信息，使得开发者能够使用 GDB/LLDB 甚至现代 IDE 源码级调试玄铁程序。
- **字节码虚拟机 (Bytecode VM)**：为 `xuantie run` 模式引入字节码中间层，替代当前的纯解释器。编译阶段将源码一次性编译为玄铁字节码，VM 执行字节码，启动速度远快于 LLVM 全编译，同时支持热重载，配合铸造厂 IDE 提供流畅的调试体验。

  ```
  xuantie run 程序.xt    → 字节码VM执行（快速启动，调试友好）
  xuantie zao 程序.xt    → LLVM编译为机器码（性能最优，用于发布）
  ```
- **原生协程调度器 (Native Coroutine Scheduler)**：自举完成后，替换当前依赖 Go 转译器的异步底层，在 xt_runtime.c 中实现完整的并发运行时。详见第10节。

## 9. 玄铁内存管理方案

### 9.1 完整四层架构

```
编译期：
  └── 第一层：环检测+自动降级
         编译器静态分析引用图
         发现确定的循环结构 → 自动把其中一条边降级为弱引用
         静默处理+输出编译警告，开发者可选择忽略或手动标注 `弱`
         零运行时开销
         ↓ 处理不了的（动态引用、跨模块复杂结构）

  └── 第二层：Arena 自动推导
         编译器分析函数/作用域内对象的生命周期
         生命周期与作用域完全对齐的对象 → 自动用 Arena 管理
         逃逸分析：返回值自动"逃逸"出 Arena，转交 ARC 管理
         ↓ 无法静态推导的动态对象

运行时：
  └── 第三层：ARC（自动引用计数）
         原子引用计数，线程安全
         处理动态生命周期的普通对象
         ↓ 漏网的循环引用

  └── 第四层：弱引用兜底
         开发者手动标注 `弱 设`
         或由编译期环检测自动插入
```

### 9.2 控制粒度分层（面向不同开发者）

玄铁支持三种内存控制粒度，开发者按需选择：

```
普通开发者（全自动）：
    默认 ARC + 编译期 Arena 推导
    完全无感知，专注业务逻辑

性能敏感开发者（半手动）：
    显式声明 Arena 区域，控制批量释放时机
    区域 {
        设 节点A = 造 节点()
        设 节点B = 造 节点()
        // 循环引用？无所谓，区域结束整块释放
    }

底层开发者（手动）：
    手动 {
        设 数据 = 分配(节点, 1024)
        // 直接操作内存
        释放(数据)
    }
    适用于渲染引擎内核、驱动、编译器等极致性能场景
```

### 9.3 并发感知的 ARC 自动升级

编译器分析对象的并发使用情况，自动选择 ARC 类型：

```
// 编译器发现此对象只在单线程使用
// → 自动用非原子 ARC，零同步开销

// 编译器发现此对象跨异步任务共享
// → 自动升级为原子 ARC，保证线程安全
```

开发者完全无感知，编译器自动推导。

### 9.4 意图感知的多策略自动切换

编译器在同一程序内对不同代码块自动选择最优内存策略：

```
// 生命周期明确的函数 → 编译器自动用 Arena
函 处理请求(请, 答) {
    设 数据 = 解(请["体"])   // Arena 分配
    设 结果 = 处理(数据)      // Arena 分配
    答.写(化(结果))
}   // 函数结束，整块释放

// 跨函数共享的对象 → 编译器自动用 ARC
设 全局缓存 = 造 缓存()

// 循环内极短命的对象 → 编译器自动栈分配，零堆开销
遍历 元素 于 列表 {
    设 临时 = 造 节点(元素)  // 栈上分配
}
```

### 9.5 编译期内存预算（嵌入式场景）

允许开发者声明内存上限，编译器静态验证，超出预算在编译期报错而非运行时崩溃：

```
内存预算 64kb {
    // 编译器静态分析此作用域最大内存占用
    // 超出 → 编译错误
}
```

### 9.6 ARC 弱引用语法

```
型 父节点 {
    设 子节点: 子节点?
}

型 子节点 {
    弱 设 父节点: 父节点?  // "弱"字一目了然，比 Swift 的 weak var 更直觉
}
```

### 9.7 玄铁自举编译器使用 Arena (v0.15.6)

自举编译器内部采用 Arena 分配所有 AST 节点，彻底规避循环引用崩溃问题：

```
自举编译器内部 → Arena（区域分配）
    已集成至 编译.xt 与 入口.xt
    编译过程中所有对象分配自动转向 Arena
    编译结束一键释放，零 ARC 开销，零循环引用崩溃

普通玄铁程序 → ARC + Arena（可选）
    普通逻辑默认使用 ARC 管理
    高性能/短生命周期场景可手动开启 Arena 一键回收
```

## 10. 玄铁并发运行时方案

### 10.1 设计目标

自举完成后，替换当前依赖 Go 转译器的异步底层，在 xt_runtime.c 中实现完整的原生并发运行时，对标 Go 协程的并发能力，同时保持玄铁的中文语义不变：

```
// 开发者永远只看到这个，底层实现透明
设 任务 = 异步 { 耗时任务() }
设 结果 = 等待(任务)

并行 {
    { 任务1() },
    { 任务2() }
}

设 消息道 = 道
异步 { 消息道.发("数据") }
设 收到 = 消息道.收()
```

### 10.2 底层架构

```
编译期：
  └── 异步函数 → 自动转换为状态机
         状态0：发起请求，挂起
         状态1：收到响应，继续
         零栈切换开销（对标 Rust async/await 原理）

运行时（xt_runtime.c）：
  └── 工作窃取调度器 (Work-stealing Scheduler)
         多个工作线程（对应 CPU 核心数）
         每个线程维护自己的任务队列
         空闲线程从繁忙线程尾部"偷"任务
         所有 CPU 核心始终保持满载

  └── 非阻塞 IO 层
         Linux → epoll
         Windows → IOCP
         封装为统一的玄铁异步 IO 接口
```

### 10.3 实现路线图

```
v1.0.0 之前：
    继续使用 Go 转译器的协程（现有方案够用）

v1.0.0 之后：
    第一步：xt_runtime.c 实现基础协程切换
            基于 ucontext (Linux) / Fiber (Windows)
            单线程先跑通

    第二步：加入 epoll/IOCP 非阻塞 IO
            让 求/听/连 真正原生异步

    第三步：实现工作窃取多线程调度器
            多核并行，对标 Go 调度器

    第四步：编译器支持状态机转换
            零栈切换，对标 Rust async
```

### 10.4 并发目标定位


| 对比项     | Go            | Rust async   | 玄铁（目标）     |
| ------------ | --------------- | -------------- | ------------------ |
| 语法复杂度 | 低            | 高           | 最低（中文语义） |
| 错误处理   | if err != nil | Result+?     | .接着/.否则链式  |
| 调度器     | 工作窃取      | 依赖第三方   | 工作窃取（内置） |
| 栈切换     | 有（分段栈）  | 无（状态机） | 无（状态机）     |
| 学习成本   | 低            | 极高         | 最低             |

### 中期可行（渲染引擎完成后）

- **铸造厂 IDE**：用玄铁渲染引擎绘制玄铁自己的 IDE，致敬 Godot 用自身引擎绘制编辑器的设计。底层对接 raylib 或类似轻量图形库，提供 `画`、`界面` 等原语。
- **Godot GDExtension 绑定**：让开发者用玄铁写 Godot 游戏逻辑，底层渲染交给 Godot，无需自建渲染引擎。"用中文写 Godot 游戏"在国内独立游戏开发者圈有真实需求。

### 长期可行（生态成熟后）

- 安卓 App（需要 JNI 或单独运行时）
- Linux 嵌入式（需要极简运行时，配合第9节内存预算机制）
- 玄铁自建渲染引擎（积累足够后，内部用手动内存控制榨取极致性能，对外暴露全自动安全 API）

### v1.2.0 TiePM 铁铺

* 开启官网tiepm.xt.markjy.com
* 只存索引不存代码-铁铺网站（索引 + 搜索）
* CLI：铁铺 安装 铁器名
* CLI：铁铺 发布
* 基础哈希验证
* 并发下载
* 为了防止版本漂移，参考 Cargo 的做法：

  * 索引里存的是每个版本的**内容哈希** ，下载时验证哈希，哈希不对就拒绝安装
  * ```
    铁铺索引记录：
      铁器名：玄铁.网络
      版本：1.2.0
      Git地址：github.com/xxx/xt-network
      提交哈希：a3f8c2d...  ← 精确到 commit，不是 tag
      内容哈希：sha256:e4b9...  ← 所有文件的哈希
    ```
* 由于会出现这样的问题：A铁器引用10个其他铁器，但是其中一个删库或访问不了，那么A铁器以及其他依赖这个铁器的项目都安装失败。所以长期解决方案是提供**镜像缓存**，国内服务器缓存热门铁器的源码包。不过这是后话，因为需要存储成本

  * 镜像缓存
  * 私有铁铺
